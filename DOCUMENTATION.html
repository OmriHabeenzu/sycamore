1 System Overview

Sycamore is a digital management platform for Sycamore Agricultural Services Cooperative — a real SACCO (Savings and Credit Cooperative Organization) based in Zambia, managed by Stalwart Services Limited. This is a single-tenant system — built exclusively for Sycamore cooperative.

PartPurposeLocal URL
Laravel APIBackend — all business logic, database, authenticationhttp://localhost:8000/api
Admin DashboardStaff/admin interface — manage members, loans, savings, reportshttp://localhost:5173
Member PortalSelf-service section for members — view accounts, apply for loans, request withdrawalshttp://localhost:5173/my
Public WebsitePublic-facing — information, news, membership applicationshttp://localhost:5174

How a SACCO Works

  • Members save a fixed amount every month (contributions)
  • All member savings form a shared pool
  • Members borrow from this pool at low interest rates (limited to 3× their savings balance)
  • Interest earned stays in the cooperative
  • At year end, profits are distributed as dividends proportional to each member's savings
  • The cooperative is member-owned — one member, one vote

6 How the System Works

User Roles & Permissions

RoleAccess Level
adminFull access to everything. Can manage staff permissions, loan products, settings. The only role that can see the Settings page and Loan Products page.
manager / loan_officer / cashier / staffAccess restricted by permission flags granted by admin. Cannot see Settings or Loan Products.
memberMember portal only (/my/*). Can view own accounts, apply for loans, request withdrawals.

Staff Permissions

Admin goes to Staff & Access to create staff accounts and set what each person can see. Each permission is a toggle:

  • Members, Contributions, Share Capital, Groups
  • Savings, Loans, Repayments, Dividends
  • Expenses, Welfare Fund, Reports, Governance
  • Inbox, News

Staff cannot see Settings (prevents self-promotion to admin) or Loan Products (interest rates are admin-only).

Authentication Flow

  1. User logs in with email + password at /login
  2. Laravel Sanctum issues a secure API token
  3. Token stored in localStorage; sent with every request as Authorization: Bearer {token}
  4. After login: members are redirected to /my, staff/admin to /

Member Lifecycle

Website Form Submitted (status = pending)
        ↓
Admin Reviews Application in Dashboard
        ↓
Orientation Conducted, Agreement Signed
        ↓
Admin Clicks "Approve" → status = active
Member Portal Account Auto-Created (email + generated password shown)
        ↓
Member Makes First Contribution + Savings Deposit
        ↓
Monthly Contributions Continue
        ↓
Member Eligible for Loans (up to 3× savings balance)
        ↓
Dividends Paid at Year End

Loan Eligibility — 3× Savings Rule

A member cannot borrow more than 3 times their total savings balance. This applies both when staff create a loan and when a member applies from the member portal. Example: if a member has K2,000 in savings, they can borrow a maximum of K6,000.

Loan Lifecycle

Applied (pending) → Approved → Disbursed → Active → [Repayments] → Closed
                  ↘ Rejected                        ↘ Overdue / Written Off
  • Interest methods: Flat Rate or Reducing Balance
  • Repayment frequencies: Daily, Weekly, Biweekly, Monthly, Quarterly
  • Schedule generation: Full amortization table created automatically on disbursement

Savings & Withdrawals

  • Each member has one or more Savings Accounts (separate from contributions)
  • Supports deposits, withdrawals, and annual interest posting
  • Withdrawal workflow: member or staff submits a request → admin/staff approves → savings balance is deducted and a transaction is recorded. Members cannot directly withdraw; all withdrawals require approval.

Welfare Fund

  • Members pay a welfare contribution per period (e.g., monthly)
  • Members or next-of-kin can submit claims (bereavement, medical, emergency, other)
  • Admin reviews and approves/rejects claims, sets the approved amount
  • Admin marks approved claims as paid when the money is disbursed

Savings Compliance

The Savings Compliance page shows which members are meeting their agreed monthly savings commitment. Staff can select any month and see a progress bar per member showing contributions received vs. their commitment amount.

Contributions vs Savings Accounts

  • Contributions — monthly SACCO savings recorded by staff (tracked against a commitment amount)
  • Savings Accounts — a flexible savings account per member with full transaction history

Dividends

At year end, admin calculates dividends based on total cooperative profit and each member's proportional savings. The system computes per-member amounts, allows review, approval, and marks as distributed once paid.

Overview cards: total members, active loans, total savings, outstanding balance, loan collection rate, overdue count. 6-month disbursement trend chart. Recent loans and latest repayments.

Applying for a Loan (Member)

  1. Go to My Loans → click Apply for Loan
  2. Select a loan product — terms are pre-filled from the product
  3. Enter amount — system shows the 3× savings maximum
  4. Enter repayment term (within the product's min/max)
  5. Optionally add a purpose/note
  6. Submit — loan created with pending status, goes to admin/staff for approval

Requesting a Withdrawal (Member)

  1. Go to My Savings
  2. Click Withdraw on the savings account
  3. Enter amount (cannot exceed current balance) and reason
  4. Submit — creates a pending withdrawal request for admin/staff to approve
  5. Status tracked in the Withdrawal Requests table below the savings accounts
Note Funds are not released immediately. All withdrawals require approval by office staff. Only one pending withdrawal request is allowed per savings account at a time.

15 Known Issues & Notes

Security — Before Going Live The APP_KEY in the repo must NOT be used in production. Run php artisan key:generate on the production server. Set APP_DEBUG=false in production — never expose Laravel debug output publicly.
Database Driver Difference (Reports) Reports use DATE_FORMAT() which is MySQL syntax. If production uses PostgreSQL (Railway), update DashboardController and ReportController to use TO_CHAR(). Test all reports after deployment.
File Storage on Deployment Profile photos and documents are stored in storage/app/public/. On Railway, the filesystem is ephemeral — files are lost on redeploy. Before go-live, configure FILESYSTEM_DISK=s3 and set up an S3-compatible bucket (AWS S3, Cloudflare R2, etc.).
CORS Configuration CORS currently allows all origins (*). Before production, update lms/config/cors.php to list only the specific frontend domains.
Member Account Email Conflicts If a member's email address is already used by a staff account, their member portal login email is automatically set to [phone]@sycamore.member. The approval modal shows the actual login email that was assigned.
Hardcoded company_id = 1 The public website endpoints hardcode company_id = 1. This is correct for Sycamore's single-tenant setup.
